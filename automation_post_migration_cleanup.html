Post-Migration Action Items

1. Set Flow Actors (All Rules)

JCMA does not migrate flow actors. Every rule needs its actor set on Cloud. Recommended: set to the Jira automation service account or the project admin.

  • Navigate to each project's Automation settings
  • Open each rule > click the actor/owner dropdown > select the appropriate user
  • For service desk projects (ITSM, SSSD, FSD): ensure the actor has agent permissions

4. Validate and Bulk-Enable Rules

All migrated rules arrive DISABLED on Cloud. Use the migration script to validate and enable in bulk:

# Step 1: Validate all rules against Cloud registries
python3 migrate_automations.py --validate \
    --target-site daybreakgames.atlassian.net \
    --token "$JIRA_CLOUD_PROD_PAT"

# Step 2: Fix any remaining ID mapping gaps
python3 migrate_automations.py --fix \
    --source-site jira.daybreakgames.com --source-is-dc \
    --target-site daybreakgames.atlassian.net \
    --dc-token "$JIRA_DC_PROD_PAT" --token "$JIRA_CLOUD_PROD_PAT"

# Step 3: Repair known manual overrides (if needed)
python3 migrate_automations.py --repair \
    --target-site daybreakgames.atlassian.net \
    --token "$JIRA_CLOUD_PROD_PAT"

# Step 4: Enable all rules that pass validation
python3 migrate_automations.py --enable-clean \
    --target-site daybreakgames.atlassian.net \
    --token "$JIRA_CLOUD_PROD_PAT"

5. Verify Project Scoping

JCMA should preserve project scoping natively. However, if any rules end up globally scoped or on the wrong project, use the auto-rescope command with the DC export JSON:

python3 migrate_automations.py --auto-rescope \
    --source-site jira.daybreakgames.com --source-is-dc \
    --target-site daybreakgames.atlassian.net \
    --dc-token "$JIRA_DC_PROD_PAT" --token "$JIRA_CLOUD_PROD_PAT" \
    --dc-rules-json ~/daybreak-scripts/PROD_EXPORT.json
